Geschichtliches zu Git

  • Entwickelt für die Quellcode-Verwaltung des einzigen Open-Source-Projektes: Linux
  • Entwicklung begann im April 2005 durch Linus Torvalds.
  • Kernel-Entwickler waren gezwungen die Verwendung des proprietaeren BitKeeper-Systems aufzugeben.

Anforderung zur Entwicklungszeit

  • Unterstuetzung verteilter Arbeitsablaeufe
  • Sehr hohe Sicherheit gegen sowohl unbeabsichtige wie auch boeswillige Verfaelschung
  • Hohe Effizienz

Weitere Eigenschaften

  • Nicht-lineare Entwicklung: Branches und Merging als feste Bestandteile.
  • Kein zentraler Server: Jeder besitzt eine lokale Kopie des gesamten Repositories.
  • Datentransfer zwischen den Repositories: SSH, HTTP(S), FTP, rsync oder Mail
  • Kryptografische Sicherheit der Projektgeschichte: Ein Commit basiert auf der Geschichte, die zu ihm gefuehrt hat.

Noch mehr Eigenschaften

  • Datenpersistenz: Geloschte Dateien, Commits und auch Branches koennen wiederhergestellt werden bis sie explizit geloescht werden.
  • Interoperabilitaet: Hilfsprogramme ermoeglichen Datenaustausch zwischen arch, SVN, CSV, Mercurial und Quilt.
Zurueck zum Titel: Git vs. Subversion

Gemeinsamkeiten

  • Source-Code-Verwaltung
  • Branching
  • Tagging
  • Wechseln zwischen Revisionen
  • Ermoeglicht zentrales Repository zum Austausch

Unterschiede

  • Git erzwingt den zentralen Abgleich nicht
  • Stattdessen: Dezentrales Revisionsmanagement
  • Arbeiten ohne Internet-Verbindung, Server-Zugang und Latenzen

Terminologie

Das Repository

  • SVN: Repository-Checkout ist eine Kopie eines Entwicklungsstandes
  • Git: Ein Repository ist eine vollstaendige Kopie aller Repository-Informationen, Commits und Branches.
  • Kann sowohl lokale als auch entfernte Branches beinhalten.

Terminologie

Die URL

  • SVN: Feste Adresse, meistens mit Unterverzeichnissen trunk/, branches/ und tags/.
  • Git: Nur die Adresse eines Repository-Verzeichnisses. Enthaelt immer Branches und Tag, aber nicht als Ordner sondern als Meta-Information.

Terminologie

Die Revision

  • SVN: Dezimalnummer
  • Git: Funktioniert nicht fuer verteilte Systeme, da jeder seine eigene Kopie weiterentwickelt und Branches unabhaengig von einem Hauptzweig sein koennen.

Mehr zu Git-Revisionen

  • Identifiziert Revisionen anhand von 160-bit-langen SHA1-Hashes.
  • Keine Angst: Lassen sich abkuerzen (5 Zeichen sind meistens eindeutig).
  • Zeiger auf die aktuelle Revision: HEAD
  • Parent: HEAD^
  • Parent's parent: HEAD^^ == HEAD~2

Terminologie

Der Commit

  • Jeder Commit gehoert einem Autor und einem Comitter.
  • Autor kann abweichen, wenn man etwa den Patch eines anderen committet.
  • Benutzerdaten einrichten:
git config --global user.name "Your Name Comes Here"
git config --global user.email you@yourdomain.example.com

Tutorial

Staging und Commit

  • add fuegt Dateien zum Staging hinzu.
  • Commit fuehrt das aus, was wir von SVN als Commit kennen.
  • Ermoeglicht Aufteilung in mehrere Commits
  • Bessere Uebersicht ueber den aktuellen Arbeitsstand

Tutorial

Schritt 2: Diff

  • Differenz zum HEAD: 
    git diff
  • Differenz der Datei [pfad] zur [revision]
    git diff [revision] -- [pfad]
  • Patch [pfad] anwenden
    git apply [pfad]
